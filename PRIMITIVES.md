# Planning primitives

The **planner** emits a *plan* — a short program the executor then follows while it reasons. The plan
is **factored** and produced by a **single autoregressive head** over a flat token sequence:

```
MODEL as=truth_table  LINK guard=on  VERIFY aspect=logic  FINALIZE form=yes_no  END
```

Every token is either a **base primitive**, a **parameter-atom** (`key=value`), or the `END`
terminator. Because primitives and parameters are separate tokens, the planner **composes** them — a
primitive+parameter pairing it never saw together is just a novel sequence of known tokens.

> **Why factored?** An earlier version made each `MODEL[as=truth_table]` one *atomic* token, which
> welds the primitive to its parameters (no composition) and collapsed 1000 problems to 18 plans.
> Splitting into sub-tokens restores compositionality with a single head — see
> [`DESIGN_CONTRASTS.md`](DESIGN_CONTRASTS.md).

> **Where it's defined:** the vocab is generated by [`traces_to_sft.py`](traces_to_sft.py) into
> `plan_vocab.json` and loaded by [`model_joint.py`](model_joint.py). The trace vocabulary below has
> **79 tokens** (PAD + 23 primitives + 54 parameter-atoms + END). With no `plan_vocab.json` present,
> `model_joint` falls back to a default 41-token bare vocab (for the legacy synthetic path).

## The 23 primitives

| Primitive | Role |
|---|---|
| `EXTRACT` | Pull out the givens (`what=` claims / numbers / quantities / relations). |
| `FRAME` | State the goal / what a complete answer needs (`want=completeness`). |
| `IDENTIFY_UNKNOWN` | Name what's being solved for (`form=` the expected answer type). |
| `DECOMPOSE` | Break the problem into parts (`by=` constraints / quantities). |
| `MODEL` | Build a representation (`as=` truth_table / state_machine / equation / rate / sequence / venn / set). |
| `LOOKUP_PROPERTY` | Check a property of the structure (`prop=` transitive / bounds). |
| `LINK` | Connect facts / relations (`guard=on` = don't chain an invalid relation). |
| `SIMULATE` | Step through a process (`depth=` deep/medium, `watch=` step / fixed_point / **escape_midcycle**). |
| `TRACE` | Walk a chain step by step (`grain=step`). |
| `CALCULATE` | Do the computation (`op=` solve / rate / count / aggregate / arithmetic / ratio). |
| `ESTIMATE` | Sanity-estimate a magnitude (`aspect=plausibility`). |
| `COMPARE` | Compare quantities / candidates (`by=quantities`). |
| `WEIGH` | Weigh competing explanations / options (`over=tradeoffs`). |
| `GENERATE` | Produce a candidate (`form=candidate`). |
| `EXPLORE` | Search the space (`via=bfs`). |
| `DIVERGE` | Branch on an ambiguity (`on=ambiguity`). |
| `CLARIFY` | Resolve an ambiguity (`on=ambiguity`). |
| `REFLECT` | Re-examine, naming the trap (`aspect=`…, `reason=` **off_by_one / naive_vs_correct / blind_chaining / contradiction**). |
| `VERIFY` | Check a specific aspect (`aspect=` logic / constraints / consistency / units / completeness / back_substitution / evidence / step). |
| `BACKTRACK` | Undo on a contradiction (`reason=contradiction`). |
| `CORRECT` | Fix an identified error (`reason=contradiction`). |
| `ITERATE` | Loop until a condition holds (`until=` consistency / constraints). |
| `FINALIZE` | Commit the answer in a form (`form=` yes_no / number_with_units / cannot_conclude / value / next_term / plan / candidate). Always last, before `END`. |

---

## The 17 parameter keys (the strategy signal)

Parameters are **not decoration** — they carry the decision that determines the answer. The trap
discriminators (`reason=naive_vs_correct`, `watch=escape_midcycle`, `prop=transitive`) are exactly
what makes a plan **load-bearing**: with them the executor avoids the trap; without them it falls in.

| Key | Values | What it selects |
|---|---|---|
| `as` | truth_table, state_machine, equation, rate, sequence, venn, set | the modeling mode |
| `aspect` | logic, step, evidence, constraints, plausibility, completeness, back_substitution, consistency, units | what to check/reflect on |
| `form` | yes_no, number_with_units, cannot_conclude, value, next_term, candidate, plan | the committed answer type |
| `reason` | off_by_one, naive_vs_correct, blind_chaining, contradiction | **the named trap to avoid** |
| `watch` | step, fixed_point, escape_midcycle | the simulation hazard to watch for |
| `op` | solve, rate, count, aggregate, arithmetic, ratio | the computation |
| `what` | claims, numbers, quantities, relations | what to extract |
| `prop` | transitive, bounds | the property to look up |
| `depth` | deep, medium | simulation depth |
| `until` | consistency, constraints | the loop stop condition |
| `by` | constraints, quantities | the decomposition axis |
| `guard` | on | block invalid chaining |
| `on` | ambiguity | the branch/clarify trigger |
| `over` | tradeoffs | what to weigh |
| `via` | bfs | the search method |
| `grain` | step | the trace granularity |
| `want` | completeness | the framing goal |

## Worked example

> "An ant in an 18-foot ditch climbs 3 ft/day, slips 1 ft/night. How many days to get out?"

```
EXTRACT what=quantities   MODEL as=state_machine   SIMULATE depth=deep watch=escape_midcycle
REFLECT aspect=step reason=off_by_one   FINALIZE form=number_with_units   END
```
The load-bearing tokens are `watch=escape_midcycle` and `reason=off_by_one` — they tell the executor
*not* to do the naive `18 ÷ (3−1)`. The executor still computes the specific number; the plan supplies
the strategy. Remove those atoms and the same executor falls for the trap. That gap (with-plan vs
no-plan accuracy) is the **ablation gap**.

## Coverage

Across all three trace sets: **23 primitives**, **17 parameter keys**, **54 parameter-atoms**, over
**50 problem families** (`sim_offbyone`, `relation_transitive`, `knaves`, `garden_path_arith`,
`causal_confound`, `conditional_flip_consequent`, …). Plan sequences are 6–17 tokens
(`plan_max_len=24`). To regenerate/inspect the vocab:
`python traces_to_sft.py --traces traces/*.jsonl --answers traces/answers_*.jsonl --out /tmp/x.jsonl`.
